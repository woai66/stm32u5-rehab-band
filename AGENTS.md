# AGENTS.md

本文件为代码审查/编码助手（如 Codex、Claude 等）提供本仓库的审查规则与偏好。
通用部分适用于任意仓库；末尾「嵌入式 / 裸机 / RTOS」一节仅嵌入式项目需要，其他项目可整节删除。

## 通用约定

- 始终使用**中文**进行审查和回复。
- 项目背景、目录结构、技术栈以仓库根目录的 `README.md` / `CLAUDE.md`（若有）为准；本文件不重复这些内容，只描述「如何审查」。

## 审查重点（按重要性，从高到低）

优先发现真正可能导致问题的缺陷，而不是风格问题：

1. 会导致崩溃或未定义行为的 bug：空指针/野指针、数组越界、未初始化变量、整数溢出、类型/符号不匹配。
2. 错误处理：函数返回值与错误码是否被检查；失败/异常路径是否安全。
3. 资源管理：内存、文件、句柄、锁、网络/串口等连接是否正确释放，有无泄漏或重复释放。
4. 并发与共享状态：多线程/中断/异步访问共享数据是否有正确同步（锁、原子、`volatile`、临界区）；有无竞态、死锁、优先级反转。
5. 边界条件：空输入、极值、缓冲区大小、循环边界、off-by-one。
6. 安全：注入、敏感信息硬编码或泄露、权限与输入校验缺失。
7. 性能：明显的低效（不必要的拷贝、重复计算、在热路径/持锁期间做重活）。

> 以上各项如不适用本仓库的语言或场景则跳过，不要为凑条目硬报。

## 审查范围

- 重点审查**本次改动涉及的、手写的业务/驱动代码**。
- 默认跳过：第三方库、SDK、自动生成的代码与配置（如 `node_modules/`、`vendor/`、`dist/`、构建产物，以及代码生成器产生的文件），除非本次改动确实发生在这些位置。

## 审查风格

- 按严重程度排序，优先指出真正可能导致 bug、安全或数据风险的问题。
- 不要因为格式、命名、注释等低价值问题刷屏（这些交给 formatter / linter）。
- 每条意见尽量给出：文件位置 + 问题原因 + 修改建议。
- 不确定时写「建议人工确认」，不要臆断。
- 没有明显高风险问题时，直接说明「未发现明显高风险问题」。
- 只输出审查意见，**不要直接修改代码**，除非被明确要求修复。

## 嵌入式 / 裸机 / RTOS 项目额外关注（非嵌入式项目可删除本节）

8. 中断与主循环/任务共享的变量是否声明 `volatile` 并有临界区保护；ISR 回调是否只做标记、不做耗时操作。
9. 中断优先级是否符合 RTOS 约束（如 FreeRTOS 的 `configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY`）后才调用 ISR 版 API。
10. 非重入库（如 newlib 的 `printf`/`malloc`）在多任务下是否有互斥保护。
11. 阻塞型驱动调用（如 HAL 的同步收发、`HAL_Delay`）在任务或持锁期间的阻塞时长是否可接受。
12. 任务/线程栈大小相对其实际用量（递归、大局部数组、浮点与 `printf` 开销）是否足够。
13. DMA 缓冲的生命周期、对齐与 cache 一致性。
14. 外设时序、复位流程，以及通信失败（I2C/SPI/UART 的 NACK、超时）的恢复路径。
15. 修改由代码生成器（如 STM32CubeMX）产生的文件时，改动是否位于 `USER CODE BEGIN/END` 区，避免重新生成时被覆盖。
